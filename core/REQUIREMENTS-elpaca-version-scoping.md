# Requirements Document: Emacs-Version-Scoped Compiled Artifact Caches (NEO)

**Version:** 1.0 **Date:** 2026-07-16 **Prepared by:** Requirements Analyst AI
(Socrates) **Status:** Draft

---

## 1. Executive Summary

NEO runs multiple named Emacs instances (`mav`, `mav1`, `neo-devel`,
`omega-neo-devel`, …), several of which point at bleeding-edge
`emacs-master-YYYY-MM-DD` builds that are periodically swapped out from under a
stable instance name. A recent incident showed the cost of this: elpaca
byte-compiled a package (`eshell-syntax-highlighting`) under one snapshot
binary, that snapshot was later swapped for a newer one, and the stale `.elc`
referenced internal helper variables generated by macro-expansion in
`easy-mmode.el` that the new binary's minor-mode machinery no longer produced
consistently. Because the affected package hooks `after-change-major-mode-hook`
globally, the resulting `void-variable` error fired on nearly every subsequent
command, making the session nearly unusable until the hook was manually removed.

Today, NEO has no mechanism that detects "the underlying Emacs binary changed"
and no systematic separation of compiled artifacts by the Emacs version that
produced them. Elpaca's `builds/` directory and NEO's own extension bytecode
cache (`neo/cache-file-path`) are flat, instance-scoped directories with no
version awareness at all. Native-comp's `eln-cache` has _partial_ version
scoping today — `early-init.el` redirects it to
`eln-cache/<instance>-emacs-<major>.<minor>/` — but that is coarser than what
this feature requires (see Section 18, open question on reconciling this).

This feature closes that gap with two complementary mechanisms: (1)
**version-scoped directories** — nesting elpaca builds, eln-cache, and NEO's own
cached bytecode under a subdirectory keyed by the Emacs version string,
underneath each instance's existing cache directory, so that switching to a
binary with a different reported version automatically gets a fresh, isolated
build tree with no explicit rebuild step; and (2) a manual escape hatch,
`neo/force-elisp-refresh`, for the residual risk the design knowingly accepts —
that two different master snapshots can report the _same_ `emacs-version` string
while being internally incompatible. Success is measured by the elimination of
stale-compiled-artifact failures like the one above, without introducing startup
latency or interactive friction for the common case.

## 2. Project Scope

### 2.1 In Scope

- Version-scoped directory layout, keyed by the `emacs-version` string, for:
  - elpaca's builds directory (`elpaca-builds-directory`, currently
    `elpaca/builds/` under `no-littering-var-directory` — see
    `devex/editors/emacs/core/neo-elpaca.el`)
  - native-comp's `eln-cache` (both the per-instance redirect currently set up
    in `devex/editors/emacs/early-init.el` via `startup-redirect-eln-cache`, and
    consideration of the global `native-comp-eln-load-path` entries)
  - NEO's own cached extension bytecode (`neo/cache-directory` /
    `neo/cache-file-path`, defined in
    `devex/editors/emacs/core/neo-early-init-utils.el`)
- All version-scoped directories nest **underneath** each instance's existing
  per-instance cache root (`~/.cache/<instance>/...`) — version scoping composes
  with, and does not replace, the existing multi-instance model.
- Automatic, silent (re)build into a version-keyed directory the first time a
  given `emacs-version` is seen for an instance — no user prompt, no startup
  block.
- Silent-failure handling: if an automatic rebuild fails, log the error and skip
  that feature/package for the session (no fallback to a stale artifact —
  loading a stale artifact is the exact failure mode this feature exists to
  prevent).
- An interactive command, `neo/force-elisp-refresh`, that wipes and rebuilds the
  _entire_ version-scoped tree (elpaca builds + eln-cache + NEO's own cached
  bytecode) for the currently running Emacs's version, as a manual escape hatch
  for same-version-string incompatibilities (see Section 4, Assumptions).
- Surfacing the on-disk size of the version-scoped cache tree in the existing
  NEO dashboard's "Neo Environment" section
  (`devex/editors/emacs/extensions/extensions/neo/dashboard/neo-dashboard.el`,
  `neo/dashboard--debug-lines`, currently populating LABEL/COMMAND pairs around
  lines 177–241), so disk usage is visible even though automatic pruning is out
  of scope.

### 2.2 Out of Scope

- **Build-fingerprint-based detection** (git commit hash /
  `emacs-repository-version`, `emacs-build-time`, or a content hash of the
  binary). Detection is based purely on the reported `emacs-version` string; the
  residual risk of same-version-string, different-snapshot incompatibility is
  explicitly accepted and mitigated only by the manual `neo/force-elisp-refresh`
  escape hatch (see Section 4).
- **Automatic pruning/cleanup of old version-scoped directories.** Stale
  version-dirs (e.g. left behind after a 30.2 → 31.0 → 32.0 progression) are not
  automatically removed in this iteration. A follow-up `bd` bead has been filed
  separately by the user to track this as future work; this document does not
  duplicate that tracking, it only records the item as an explicit non-goal.
- Version-scoping of elpaca's **package sources** (`elpaca-sources-directory` /
  git checkouts under `elpaca/repos/`) — source code is not version-dependent,
  only compiled artifacts are, so sources remain shared across all Emacs
  versions for a given instance.
- Interactive prompting or startup blocking on mismatch — all detection and
  rebuild in the automatic path is silent.
- Finer-grained refresh commands (e.g. "refresh only elpaca" or "refresh only
  eln-cache"); `neo/force-elisp-refresh` is a single all-or-nothing command for
  this iteration.
- Any change to the `setup.sh` / Ansible `emacs-install` role that produces the
  `emacs-master-YYYY-MM-DD` binaries themselves.

### 2.3 Assumptions

- **[Assumption]** Two different Emacs master snapshots that happen to report
  the same `emacs-version` string are treated as _compatible_ by default. This
  is a deliberate risk acceptance by the user, not an oversight — the
  alternative (fingerprinting by build time/commit/binary hash) was explicitly
  rejected as unnecessary complexity for this iteration. The
  `neo/force-elisp-refresh` command exists specifically to cover the case where
  this assumption turns out false for a given snapshot swap.
- **[Assumption]** Because master-snapshot users' reported `emacs-version`
  typically does not change between snapshot swaps (per the incident that
  motivated this feature), the automatic version-scoped-directory mechanism
  will, in practice, rarely create a _new_ directory for those users on a
  snapshot swap — it primarily protects users who move between distinct version
  strings (e.g. stable version upgrades, or an instance name repointed from one
  major/minor version to another). The manual escape hatch is therefore the
  primary defense for master-snapshot users specifically, not the automatic
  directory versioning. This asymmetry should be visible to users (e.g. in
  documentation or dashboard framing) so expectations are set correctly.
- **[Assumption]** The existing eln-cache redirect in `early-init.el` (keyed by
  `<instance>-emacs-<major>.<minor>>`) will be reconciled with the new, more
  granular `emacs-version`-string-based scheme by the
  architecture/implementation phase — see Section 18.

### 2.4 Constraints

- Must compose with, not replace, the existing per-instance directory model
  (`neo/get-emacs-instance-name` → `~/.cache/<instance>/...`).
- Must not perceptibly increase Emacs startup latency for the common case (i.e.,
  when the current version's directory already exists and is populated) — see
  Section 7.1.
- Elisp implementation must live under `devex/editors/emacs/core/` (where
  `neo-elpaca.el` and `neo-early-init-utils.el` already live) and follow NEO's
  `neo/` namespacing conventions.

## 4. Business Requirements

### 4.1 Business Objectives

- **BR-001:** Eliminate session-breaking failures caused by loading
  byte-compiled/native-compiled artifacts that were built against a different
  Emacs binary than the one currently running.
- **BR-002:** Make disk usage of version-scoped compiled-artifact caches visible
  to the user without requiring manual filesystem inspection, in lieu of
  automatic cleanup (deferred to a separate tracked follow-up).
- **BR-003:** Preserve the zero-friction startup experience NEO currently
  provides — no new prompts or blocking steps in the common case.

### 4.2 Success Metrics

- **KPI-001:** Zero recurrences of the `void-variable`-style stale-artifact
  failure class after a genuine `emacs-version` change, measured by absence of
  related incidents/bug reports.
- **KPI-002:** Manual recovery from a same-version-string snapshot
  incompatibility takes exactly one command (`neo/force-elisp-refresh`) with no
  other manual cache-directory surgery required.
- **KPI-003:** No measurable startup-time regression when the current version's
  directory is already warm (see NFR in Section 7.1).

## 5. User Requirements

### 5.1 User Personas

**Persona 1: NEO maintainer running multiple instances**

- Runs several named Emacs instances (`mav`, `mav1`, `neo-devel`,
  `omega-neo-devel`) concurrently, some pinned to stable Emacs releases, others
  tracking `emacs-master` snapshots that are rebuilt/replaced periodically
  outside of NEO's control.
- Goals: never have a snapshot swap silently corrupt a working session; be able
  to recover quickly and predictably when it does happen; not be interrupted by
  prompts during normal startup.
- Pain points (from the motivating incident): a stale `.elc` referencing
  version-specific internal variables caused a globally-hooked package to error
  on nearly every command, degrading the session to near-unusable before the
  root cause (a stale build under an old binary) was identified and manually
  worked around.

### 5.2 User Stories

**Epic 1: Version-isolated compiled artifacts**

- **US-001:** As the NEO maintainer, I want elpaca builds, eln-cache, and NEO's
  own cached bytecode to live in a directory keyed by the running Emacs's
  version string, so that switching to a binary with a different version never
  loads compiled code built for a different one.
  - Acceptance Criteria: Given an instance previously run under Emacs version A,
    when it is next started under Emacs version B (B ≠ A), then elpaca builds,
    eln-cache, and NEO's cached bytecode are all (re)built fresh under a
    version-B-keyed directory, with no artifacts from version A read or reused.
  - Priority: Must Have

- **US-002:** As the NEO maintainer, I want the (re)build on first use of a new
  version's directory to happen automatically and silently, so that a normal
  version bump doesn't interrupt my startup with a prompt.
  - Acceptance Criteria: Given a version-keyed directory does not yet exist for
    the current `emacs-version`, when Emacs starts, then the directory is
    created and required artifacts are built without any interactive prompt or
    blocking dialog.
  - Priority: Must Have

- **US-003:** As the NEO maintainer, I want a single command to force a full
  rebuild of the current version's compiled-artifact tree, so that I can recover
  when two master snapshots share a version string but are actually
  incompatible.
  - Acceptance Criteria: Given `M-x neo/force-elisp-refresh` is invoked, when it
    completes, then the elpaca builds, eln-cache, and NEO cached bytecode
    directories for the current `emacs-version` have been wiped and rebuilt from
    (shared, non-version-scoped) sources.
  - Priority: Must Have

- **US-004:** As the NEO maintainer, I want to see how much disk space the
  version-scoped cache tree is using from the dashboard, so that I'm not
  surprised by accumulation even though nothing prunes it automatically yet.
  - Acceptance Criteria: Given the NEO dashboard is opened, when the "Neo
    Environment" section renders, then it includes a line showing the on-disk
    size of the current instance's version-scoped cache tree.
  - Priority: Must Have

- **US-005:** As the NEO maintainer, when an automatic rebuild fails (e.g. a
  package fails to compile against the new Emacs version), I want that failure
  logged and the affected feature skipped for the session, rather than silently
  falling back to a known-stale artifact.
  - Acceptance Criteria: Given a package/artifact fails to build during the
    automatic version-directory-populate step, when startup continues, then the
    failure is logged (visible in `*Messages*` or equivalent), the affected
    feature/package is not loaded for that session, and no stale artifact from a
    different version directory is substituted.
  - Priority: Must Have

## 6. Functional Requirements

### 6.1 Version-Scoped Directory Layout

**FR-001: Version-keyed subdirectory for elpaca builds**

- Description: `elpaca-builds-directory` resolves to a path nested under the
  instance's existing elpaca directory, further nested under a subdirectory
  named/keyed by the current `emacs-version` string.
- Inputs: `emacs-version`, `neo/get-emacs-instance-name`,
  `no-littering-var-directory` (per existing redirect in `neo-elpaca.el`).
- Processing: On Emacs startup, before elpaca's build directory is first used,
  compute the version-keyed path and (re)bind `elpaca-builds-directory` to it.
- Outputs: elpaca builds land under
  `~/.cache/<instance>/elpaca/builds/<emacs-version>/` (exact path shape is a
  design decision for @vitruvious/@vulcan-elisp).
- Business Rules: `elpaca-sources-directory` is explicitly **not**
  version-scoped (Section 2.2).
- Priority: Must Have
- Dependencies: FR-004 (version-string retrieval)

**FR-002: Version-keyed subdirectory for eln-cache**

- Description: Native-comp's eln-cache is scoped by `emacs-version` in addition
  to the existing per-instance scoping.
- Inputs: `emacs-version`, existing `startup-redirect-eln-cache` call in
  `early-init.el` (currently keyed by `<instance>-emacs-<major>.<minor>`).
- Processing: Reconcile the existing major.minor-based redirect with the
  full-`emacs-version`-string scheme required here (see Section 18, open
  question — this reconciliation is a design decision, not resolved by this
  document).
- Outputs: eln-cache artifacts isolated per `emacs-version` per instance.
- Business Rules: Global `native-comp-eln-load-path` entries outside NEO's
  control (e.g. the Emacs install's own `native-lisp/`) are read-only
  system-provided `.eln` files and are out of scope for version-scoping (they
  already match their own binary by construction).
- Priority: Must Have
- Dependencies: FR-004

**FR-003: Version-keyed subdirectory for NEO's own cached bytecode**

- Description: `neo/cache-directory` / `neo/cache-file-path` (defined in
  `neo-early-init-utils.el`) gain a version-scoped segment for any cached
  compiled elisp (e.g. extension bytecode) they store.
- Inputs: `emacs-version`, `neo/cache-directory`.
- Processing: Extend `neo/cache-file-path` (or introduce a version-scoped
  variant) so callers that cache compiled artifacts write under the
  version-keyed subdirectory.
- Outputs: NEO's own compiled-artifact cache entries isolated per
  `emacs-version`.
- Business Rules: Non-compiled cached data (e.g. extension manifests, registry
  state) is not necessarily version-dependent and should not be forced into the
  version-scoped tree unless it actually contains compiled code — exact boundary
  is a design decision.
- Priority: Must Have
- Dependencies: FR-004

**FR-004: Version-string retrieval**

- Description: A single, shared function/value NEO uses everywhere it needs the
  version fingerprint, returning the value of `emacs-version` (or an equivalent,
  sanitized-for-filesystem-use form of it).
- Business Rules: No build-time, commit-hash, or binary-hash fingerprinting
  (Section 2.2) — `emacs-version` string equality is the sole criterion for
  "same version."
- Priority: Must Have
- Dependencies: None

### 6.2 Automatic Build-on-First-Use

**FR-005: Silent build into a new version directory**

- Description: When Emacs starts and the version-keyed directory for the current
  `emacs-version` does not yet exist (or is missing required artifacts), the
  relevant subsystem (elpaca / native-comp / NEO cache) builds into it
  automatically, without prompting.
- Business Rules: No interactive confirmation; no startup block pending full
  completion (async/background build is acceptable per existing
  elpaca/native-comp async behavior).
- Priority: Must Have
- Dependencies: FR-001, FR-002, FR-003

**FR-006: Rebuild-failure handling**

- Description: If a build into a version-keyed directory fails for a given
  package/artifact, the failure is logged and that package/feature is skipped
  for the session.
- Business Rules: No fallback to an artifact from a different version's
  directory; no interactive prompt.
- Priority: Must Have
- Dependencies: FR-005

### 6.3 Manual Escape Hatch

**FR-007: `neo/force-elisp-refresh` command**

- Description: An interactive command that wipes and rebuilds the entire
  version-scoped tree (elpaca builds + eln-cache + NEO cached bytecode) for the
  Emacs version currently running.
- Inputs: None (operates on the current session's `emacs-version`).
- Processing: Delete the current version-keyed directories for all three
  artifact classes in scope, then trigger a fresh build of each from their
  respective (shared, non-version-scoped) sources.
- Outputs: A fully rebuilt version-scoped tree for the current version.
- Business Rules: Single all-or-nothing scope for this iteration (Section 2.2) —
  no per-artifact-class variants.
- Priority: Must Have
- Dependencies: FR-001, FR-002, FR-003

### 6.4 Dashboard Visibility

**FR-008: Version-scoped cache size in "Neo Environment"**

- Description: The dashboard's "Neo Environment" section (`neo-dashboard.el`,
  `neo/dashboard--debug-lines`) gains a new LABEL/COMMAND-style line showing the
  on-disk size of the current instance's version-scoped cache tree (elpaca
  builds + eln-cache + NEO cached bytecode, for the currently running version —
  or, alternatively, summed across all version-dirs for the instance; exact
  scope is a design decision, see Section 18).
- Business Rules: Read-only informational line; no action bound to it in this
  iteration (unlike, e.g., the "Package conflicts" line which is interactive).
- Priority: Must Have
- Dependencies: FR-001, FR-002, FR-003

## 7. Non-Functional Requirements

### 7.1 Performance Requirements

- Computing the version-fingerprint (`emacs-version` string) and resolving
  version-scoped paths must add no perceptible startup latency — this is a
  string comparison and path computation, not I/O-bound work, and must not,
  e.g., hash the binary or scan directories synchronously during startup.
- When a version directory already exists and is populated (the common case for
  a stable-version user, or a master-snapshot user per the accepted-risk
  assumption in Section 2.3), startup performance must be indistinguishable from
  today's baseline.

### 7.2 Scalability Requirements

- [Not applicable — single-user, local-machine feature.]

### 7.3 Availability & Reliability

- A failed rebuild for one package/artifact (FR-006) must not prevent Emacs from
  starting or block unrelated packages/artifacts from loading.

### 7.4 Security Requirements

- [Not applicable beyond existing elpaca/NEO trust model — no new network or
  credential surface introduced by this feature.]

### 7.5 Usability Requirements

- Zero interactive prompts in the automatic path (US-002).
- `neo/force-elisp-refresh` must be discoverable via `M-x` with a
  self-explanatory name and docstring explaining what it wipes/rebuilds and why
  one would invoke it (same-version-string snapshot incompatibility).

### 7.6 Maintainability Requirements

- Version-fingerprint retrieval (FR-004) should be a single shared utility, not
  duplicated across the elpaca, eln-cache, and NEO-cache code paths, to avoid
  the three subsystems drifting out of sync on what "version" means.

### 7.7 Compatibility Requirements

- Must continue to work across the range of Emacs versions NEO already supports
  (30.2+ per `README.md`), including both stable releases and `emacs-master`
  snapshot builds.
- Must not regress the existing per-instance model
  (`neo/get-emacs-instance-name`) for users not affected by version swaps.

## 8. Data Requirements

### 8.1 Data Entities

- **Version-scoped directory**: a filesystem path segment keyed by
  `emacs-version`, nested under each of: elpaca builds, eln-cache, NEO cached
  bytecode, in turn nested under the per-instance cache root.
- **Cache-size metric**: a computed value (bytes, likely human-formatted)
  representing on-disk usage of the version-scoped tree, surfaced on the
  dashboard.

### 8.2 Data Quality Requirements

- [TBD — exact path-sanitization rules for embedding `emacs-version` (which may

  retention/pruning policy. Old version-dirs persist indefinitely until

## Appendices

### Appendix A: Glossary

- **elpaca**: NEO's package manager (github.com/progfolio/elpaca), whose
  `builds/`/`sources/` directories are redirected under
  `no-littering-var-directory` in `neo-elpaca.el`.
- **eln-cache**: Emacs native-comp's cache of `.eln` native-compiled files,
  configured via `native-comp-eln-load-path` and `startup-redirect-eln-cache`.
- **Instance**: A named Emacs configuration profile, derived by
  `neo/get-emacs-instance-name` from the `--name`/`-name` flag, `$EMACS_NAME`,
  or `neo/default-emacs-instance-name`; each instance gets its own
  `~/.cache/<instance>/...` tree.
- **Version-scoped directory**: A subdirectory keyed by the `emacs-version`
  string, nested under an instance's cache tree, introduced by this feature.
